## Overview

v0.1 keeps three categories of OAuth2 state in process memory: pending
authorization requests, short-lived authorization codes, and the JWT
revocation blocklist. It also has no concept of tenant â€” projects are owned
directly by users. This makes the server non-restartable during an active
login flow, unsafe to run as more than one instance, and hard to evolve
toward a shared-org model.

v0.2 makes every server process stateless. All transient OAuth2 state moves
to Postgres, the data model gains an `orgs` tenant boundary, schema
migrations take an advisory lock, and the HTTP server shuts down gracefully.
The end state: any instance can serve any request, rolling restarts are
safe during active flows, and horizontal scale needs nothing more than
"start another process pointing at the same database".

OTel propagation through the GitHub HTTP client landed alongside this work
and is not re-described here.

This spec covers persistence, statelessness, and the orgs migration. The
refresh token grant handler that depends on the new grants columns is
specified in `spec/refresh_tokens.md`.

## What "stateless" means here

A server process is **stateless** if all three are true:

1. Any request can be served by any process â€” no sticky sessions, no
   per-instance affinity, no in-memory caches that change correctness.
2. Killing a process at any moment loses no committed work and no in-flight
   user-visible flow that would otherwise have succeeded.
3. Starting a fresh process requires only the configured secrets and the
   shared database; no warm-up, no state replay.

The remaining in-process state after v0.2 is limited to:

- The pre-marshalled JWKS document and discovery metadata, computed once at
  startup from static config.
- The signing key (`jwk.Key`), loaded once from a file or secret.
- The pgx connection pool.
- The OTel exporter clients.

None of these change at runtime, so they are safe to duplicate across
instances.

## Current in-memory state

Three maps live on the `oidc.Server` struct:

| Field | Key | Value | TTL | Purpose |
|-------|-----|-------|-----|---------|
| `pending` | GitHub OAuth state (UUID) | `*pendingAuth` | 10 min | Carries PKCE params and client redirect URI across the GitHub redirect |
| `codes` | Authorization code (UUID) | `*pendingCode` | 5 min | Carries identity and GitHub tokens waiting for the token exchange |
| `revoked` | JWT `jti` (UUID) | `time.Time` (expiry) | Until JWT `exp` | Prevents use of logged-out tokens |

All three are protected by separate mutexes. A server restart clears them
all; any in-flight login is invalidated and any recently revoked token
becomes temporarily valid again. Two instances behind a load balancer
cannot share any of this state.

## Pending authorizations

Stores the client's PKCE parameters and redirect URI during the GitHub
OAuth2 redirect leg (between `GET /oauth2/authorize` and
`GET /auth/github/callback`).

### Fields

| Column | Type | Notes |
|--------|------|-------|
| `state` | TEXT (PK) | Random UUID generated by the server; sent as GitHub `state` parameter |
| `client_id` | TEXT | Registered OAuth2 client ID from the authorize request; carried through to the grant row |
| `redirect_uri` | TEXT NOT NULL | Client's registered callback URL |
| `scope` | TEXT NOT NULL | Requested scope string |
| `code_challenge` | TEXT NOT NULL | PKCE S256 challenge |
| `client_state` | TEXT | Opaque state from the client, echoed back on redirect |
| `expires_at` | TIMESTAMPTZ NOT NULL | `now() + 10 minutes` at insert time |

### Operations

**Store** (`StorePendingAuth`): INSERT with an `expires_at` 10 minutes in
the future. Lazy-prune expired rows on each insert (one
`DELETE WHERE expires_at < now()` in the same transaction).

**Consume** (`ConsumePendingAuth`): Atomic `DELETE â€¦ WHERE state = $1 AND
expires_at > now() RETURNING â€¦`. Returns `ErrNotFound` if the row does not
exist or has expired. The atomic delete ensures a state value cannot be
reused even under concurrent requests served by different instances.

## Authorization codes

Stores identity and GitHub App tokens between the GitHub callback and the
token exchange. The authorization code has a 5-minute TTL.

This table contains sensitive data (GitHub access and refresh tokens) and
uses the same NaCl secretbox encryption as the `grants` table. An empty
token string is stored as zero bytes without invoking the encryptor, so
test environments without an encryption key can store codes that have no
GitHub tokens.

### Fields

| Column | Type | Notes |
|--------|------|-------|
| `code` | TEXT (PK) | Random UUID authorization code |
| `sub` | TEXT NOT NULL | GitHub user ID as decimal string |
| `email` | TEXT NOT NULL | Verified primary email |
| `scope` | TEXT NOT NULL | Granted scope |
| `code_challenge` | TEXT NOT NULL | PKCE S256 challenge for verification at token exchange |
| `redirect_uri` | TEXT NOT NULL | Must be echoed back by the client at token exchange |
| `client_id` | TEXT | Carried from `pending_auths.client_id` |
| `access_token` | BYTEA NOT NULL | GitHub access token, NaCl secretbox encrypted (empty bytes if absent) |
| `refresh_token` | BYTEA NOT NULL | GitHub refresh token, NaCl secretbox encrypted (empty bytes if absent) |
| `access_token_expiry` | TIMESTAMPTZ | NULL when no GitHub App token present |
| `refresh_token_expiry` | TIMESTAMPTZ | NULL when no GitHub App token present |
| `expires_at` | TIMESTAMPTZ NOT NULL | `now() + 5 minutes` at insert time |

### Operations

**Store** (`StoreAuthCode`): INSERT with `expires_at = now() + 5 minutes`.
Lazy-prune all expired auth codes (`DELETE WHERE expires_at < now()`) in
the same transaction.

**Consume** (`ConsumeAuthCode`): Atomic `DELETE â€¦ WHERE code = $1 AND
expires_at > now() RETURNING â€¦`. Returns `ErrNotFound` for unknown or
expired codes. Single-use by definition â€” the DELETE ensures a code cannot
be exchanged twice, even when two instances race the same redeem.

## Revoked tokens

Stores JWT IDs (`jti`) that have been revoked via logout or token rotation.
`VerifyJWT` checks this table on every request. Rows are self-pruning: any
row whose `expires_at` is in the past is irrelevant (the JWT has expired
regardless) and is cleaned up lazily.

### Fields

| Column | Type | Notes |
|--------|------|-------|
| `jti` | TEXT (PK) | JWT ID from the `jti` claim |
| `expires_at` | TIMESTAMPTZ NOT NULL | Mirrors JWT `exp`; entries past this time are pruned |

### Operations

**Revoke** (`RevokeToken`): INSERT with `ON CONFLICT (jti) DO NOTHING`
(idempotent). Lazy-prune expired rows (`DELETE WHERE expires_at < now()`)
in the same statement batch.

**Check** (`IsTokenRevoked`): `SELECT EXISTS(â€¦ WHERE jti = $1 AND
expires_at > now())`. The `expires_at > now()` filter means expired entries
do not affect correctness even if not yet pruned.

`IsTokenRevoked` is on the hot path (every authenticated MCP request). The
PK lookup is O(1) and the table is small (one row per active revocation),
so no caching layer is added. If this becomes a bottleneck, an in-process
LRU with a short TTL is the natural next step â€” but it introduces a
revocation-propagation window and is deferred until measured.

## Grants table additions

The `grants` table gains two columns required by the refresh token grant
spec.

```sql
ALTER TABLE grants
    ADD COLUMN our_refresh_token TEXT,
    ADD COLUMN client_id         TEXT;

CREATE UNIQUE INDEX grants_our_refresh_token_idx
    ON grants (our_refresh_token)
    WHERE our_refresh_token IS NOT NULL;
```

The columns are nullable strictly because grants may exist without a
refresh token in test/CLI scenarios (no GitHub App refresh issued). The
refresh handler treats a null `our_refresh_token` as "this grant cannot be
refreshed".

The `store.Grant` struct gains corresponding fields: `OurRefreshToken` and
`ClientID`. These are populated by the token handler when issuing a new JWT
and stored via `UpsertGrant`.

New grant operations (used by the refresh token handler):

**`GetGrantByRefreshToken`**: Looks up a grant by `our_refresh_token`.
Decrypts `access_token` and `refresh_token` before returning. Returns
`ErrNotFound` if no matching row exists.

**`RotateGrant`**: Atomic compare-and-swap update. Updates the grant row
identified by `OldOurRefreshToken` with a new JTI, new opaque refresh
token, and new GitHub tokens. Implemented as a single
`UPDATE â€¦ WHERE our_refresh_token = $old RETURNING â€¦`. Returns `ErrNotFound`
if the old token was already consumed (concurrent refresh race). Because
the PK (`jti`) is updated in place, no foreign keys on other tables
reference it.

**`DeleteGrant`**: Removes a grant row by `jti`. Called when the GitHub
refresh token is found to be expired and the session must be fully
invalidated.

## Tenancy: orgs and membership

v0.2 introduces an org tenant boundary between users and projects. Every
project belongs to exactly one org; every user belongs to one or more orgs;
facts are scoped (transitively, via project) to an org. This is the
permanent model â€” there is no production data to migrate, so v0.1's
`projects.owner_id` column is replaced in place rather than coexisting with
the new `org_id`.

### Model

- **Personal org.** Created automatically the first time a user logs in.
  Slug defaults to the GitHub login; if that slug is already taken by an
  existing shared org, fall back to `{login}-{github_id}`. The user is the
  sole `owner` member.
- **Shared org.** Created explicitly via `org_create`. Other users join via
  invitation. Roles are `owner`, `admin`, `member`.
- **Project.** Belongs to one org. Slug is unique within the org. Created
  by any member; deletion is owner/admin only.
- **Fact.** Scoped to a project (and therefore transitively to an org). No
  separate `org_id` column on facts â€” joining through `projects` is
  cheap and avoids denormalisation drift.

### Tables

```sql
CREATE TABLE orgs (
    id         UUID        PRIMARY KEY DEFAULT uuidv7(),
    slug       TEXT        NOT NULL UNIQUE,
    name       TEXT        NOT NULL,
    kind       TEXT        NOT NULL CHECK (kind IN ('personal', 'shared')),
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE org_members (
    org_id  UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    role    TEXT NOT NULL CHECK (role IN ('owner', 'admin', 'member')),
    joined_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (org_id, user_id)
);

CREATE INDEX org_members_user_idx ON org_members (user_id);
```

### Changes to existing tables

- `projects.owner_id UUID NOT NULL REFERENCES users(id)` â†’ removed.
- `projects.org_id UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE` â†’ added.
- `projects.created_by UUID NOT NULL REFERENCES users(id)` â†’ added (audit only,
  not used for access control).
- The `projects_owner_slug_unique` constraint becomes
  `projects_org_slug_unique UNIQUE (org_id, slug)`.

Because there is no production data, this is applied by editing
`migrations/1_initial_schema.sql` in place rather than as a new migration.
The end state of migration 1 is shown in [Database schema](#database-schema).

### Personal org creation

`UpsertUser` becomes a single transaction:

1. INSERT into `users` ON CONFLICT (github_id) DO UPDATE â€” returns the row
   (existing or new) and a `created` flag.
2. If `created`: INSERT into `orgs` with `kind='personal'`,
   `slug=COALESCE(login, login||'-'||github_id)` (collision-safe), then
   INSERT into `org_members` with role `owner`.
3. Both inserts use ON CONFLICT DO NOTHING so concurrent first-logins
   from the same user (rare but possible across instances) are idempotent.

The store interface changes from `UpsertUser(ctx, githubID, email, login)
error` to `UpsertUser(ctx, â€¦) (*User, error)` so the handler can pass the
internal user ID into the JWT `sub` claim. (See SPEC.md â€” `sub` becomes the
internal UUID rather than the GitHub numeric ID, now that `users.id` exists
and personal orgs need a stable internal identifier.)

### Access control

Project access is gated by org membership. `resolveProjectAccess(user,
project)` checks for a row in `org_members` matching the project's
`org_id`. Every MCP tool that takes a `project` slug also takes an optional
`org` slug; if omitted, the user's personal org is assumed. This keeps the
single-user UX unchanged while supporting shared orgs explicitly.

The exact MCP surface for org operations (`org_create`, `org_list`,
`org_invite`, etc.) is out of scope for this spec and is added in SPEC.md
under the v0.2/v0.3 roadmap.

## Schema migration locking

`Migrate` currently runs migrations one by one with no cross-process
coordination. Two instances starting simultaneously can race the same
migration, producing duplicate-key errors on `schema_migrations` (harmless)
or partial application of statements that aren't idempotent (less
harmless).

v0.2 wraps migration runs in a Postgres advisory lock:

```sql
SELECT pg_advisory_lock($1);  -- $1 = a constant project-specific bigint
-- run pending migrations
SELECT pg_advisory_unlock($1);
```

The lock key is a constant chosen at compile time (e.g.
`hash('starlogz-migrations')` truncated to int64). Only one process holds
the lock at a time; the others block in `pg_advisory_lock` until the
holder finishes, then re-check `schema_migrations` and exit immediately
because no migrations are pending.

Lock acquisition uses a context with a configurable timeout (default 60s)
so a stuck migration in one instance does not block boot indefinitely
elsewhere â€” the request fails fast and the operator investigates.

## Graceful shutdown

The HTTP command currently calls `srv.ListenAndServe()` and exits hard on
any signal. v0.2 wires SIGTERM/SIGINT to a graceful shutdown sequence:

1. Stop accepting new connections (`http.Server.Shutdown(ctx)`).
2. Wait for in-flight requests to drain, up to a deadline (default 30s,
   override via `--shutdown-timeout`).
3. Close the pgx pool.
4. Flush and shut down OTel exporters (`tracerProvider.Shutdown(ctx)`).
5. Exit.

The 30-second default is chosen to comfortably exceed an MCP `initialize`
+ `tools/call` round trip (typically < 5s) while staying inside the
30-second SIGKILL grace period that Kubernetes and most container
runtimes give by default. Operators with longer-running tool calls can
extend it.

This matters for rolling deployments: an instance receiving SIGTERM
during a tool call must finish that call rather than 502-ing the agent.
The orchestrator routes new requests to other instances during the drain.

## JWKS and signing key distribution

The signing key (`jwk.Key`) is loaded once from `--jwk-path` at startup
and never changes during the process lifetime. For multi-instance
deployments, **all instances must load the same key file**. This is a
deployment concern, not a code concern: distribute the key via the same
secret-management path that delivers `GITHUB_CLIENT_SECRET` and
`TOKEN_ENCRYPTION_KEY`.

Key rotation in v0.2 is **not zero-downtime**: changing the key requires
deploying the new file, restarting all instances, and accepting that any
JWT issued by the old key becomes invalid. Clients re-authenticate. For a
7-day JWT TTL this is rarely needed.

A multi-key JWKS scheme (one signing key, additional verification-only
keys, hot rotation) is planned for v0.3 once the operational need is
demonstrated.

## Store interface changes

Two new narrow interfaces are defined in the `oidc` package and satisfied
by `store.Store` / `*postgres.Store`:

```go
// AuthStateStore persists transient OAuth2 authorization state.
type AuthStateStore interface {
    StorePendingAuth(ctx context.Context, state string, p store.PendingAuth) error
    ConsumePendingAuth(ctx context.Context, state string) (*store.PendingAuth, error)
    StoreAuthCode(ctx context.Context, code string, c store.AuthCode) error
    ConsumeAuthCode(ctx context.Context, code string) (*store.AuthCode, error)
}

// RevocationStore persists revoked JWT IDs.
type RevocationStore interface {
    RevokeToken(ctx context.Context, jti string, expiresAt time.Time) error
    IsTokenRevoked(ctx context.Context, jti string) (bool, error)
}
```

The existing `GrantStore` interface in `oidc` is expanded with the three
new grant operations above.

`oidc.Config` gains two new required fields, `AuthState AuthStateStore`
and `Revocation RevocationStore`, replacing the three in-memory maps
removed from `oidc.Server`. These are required (nil means the relevant
flows will error at runtime) rather than optional â€” there is no
non-Postgres path for these in v0.2.

The `oidc.Server` struct loses: `revokedMu`, `revoked`, `pendingMu`,
`pending`, `codesMu`, `codes`. It gains: `authState AuthStateStore`,
`revocation RevocationStore`.

## v0.2 constraints

- **No background pruning** â€” expired rows in `pending_auths`, `auth_codes`,
  and `revoked_tokens` are pruned lazily on each insert to the same table.
  In low-traffic deployments, expired rows may accumulate between inserts.
  A scheduled prune job is planned for v0.3.
- **Encryption key required in production** â€” `auth_codes` stores encrypted
  GitHub tokens. The `TOKEN_ENCRYPTION_KEY` env var is already required for
  the grants table; no new env var is needed.
- **Signing key rotation requires restart and re-auth** â€” see
  [JWKS and signing key distribution](#jwks-and-signing-key-distribution).
  Multi-key JWKS is planned for v0.3.
- **Org membership is flat** â€” no nested orgs, no per-project roles.
  Roles are coarse (`owner`/`admin`/`member`) and apply org-wide.
